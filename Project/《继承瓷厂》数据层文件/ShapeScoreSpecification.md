## 1. 项目说明

### 1.1 评估目标

ShapeScore 用于评估：**玩家器型 vs 标准器型** 之间的匹配程度。

回答核心问题：**器型做得像不像？**

### 1.2 对应玩法

五控制点拖拽拉坯系统：

```
       口 (Mouth)
      /         \
     /   颈       \
    |   (Neck)     |
     \   肩        /
      \ (Shoulder)/
       \  腹      /
        \ (Belly)/
         \  足   /
          (Foot)
```

玩家通过拖拽 5 个控制点调整口、颈、肩、腹、足的轮廓比例，目标是尽可能接近订单指定的器型模板。

### 1.3 数据字段

| 字段 | 来源 Sheet | 作用 | 取值范围 |
|------|-----------|------|----------|
| ShapeID | ShapeConfig | 器型唯一标识 | SHAPE_001~005 |
| NameCN/NameEN | ShapeConfig | 显示名称 | 碗/盘/梅瓶/玉壶春瓶/罐 |
| MouthRatio | ShapeConfig | 口部比例 | 0~1 |
| NeckRatio | ShapeConfig | 颈部比例 | 0~1 |
| ShoulderRatio | ShapeConfig | 肩部比例 | 0~1 |
| BellyRatio | ShapeConfig | 腹部比例 | 0~1 |
| FootRatio | ShapeConfig | 足部比例 | 0~1 |
| Difficulty | ShapeConfig | 难度等级(1~5) | 不影响ShapeScore |
| UnlockLevel | ShapeConfig | 解锁等级 | 不影响评分 |

### 1.4 标准器型模板

## 2. 数学模型

### 2.1 单项误差

```
Error_k = | P_k − T_k |

P_k = 玩家输入的 k 部位比例值
T_k = 目标器型的 k 部位模板值
k ∈ {Mouth, Neck, Shoulder, Belly, Foot}
```

### 2.2 权重分配

| 部位 | 权重 | 解释 |
|------|------|------|
| Mouth (口) | **15%** | 辅助特征：碗/盘口大，梅瓶口小。限制性较强但不独立决定器型 |
| Neck (颈) | **15%** | 限定特征：仅梅瓶(0.10)和玉壶春(0.55)有颈，碗/盘颈=0 |
| Shoulder (肩) | **30%** | 核心特征：梅瓶的标志(0.85)，决定器型上部轮廓走向 |
| Belly (腹) | **30%** | 核心特征：器型最大直径位置，罐(0.80)腹鼓、盘(0.30)腹浅 |
| Foot (足) | **10%** | 辅助特征：各种器型差异较小(0.35~0.60)，区分度最低 |

### 2.3 权重合理性分析

**为什么肩部与腹部权重最高（各30%）？**

从陶瓷器型鉴定角度：

1. **肩部是器型"指纹"**：肩部比例直接决定器型上部轮廓是陡峭（碗/盘，肩0）、丰硕（梅瓶，肩0.85）还是适中（玉壶春，肩0.45）。陶瓷鉴定专家首先看肩部判断器型大类。

2. **腹部决定器型"体量"**：腹部是器型最大直径处，决定了整体胖瘦比例。罐腹鼓(0.80)、盘腹浅(0.30)，差异悬殊。腹部+肩部联合基本锁定器型身份。

3. **口颈足的辅助性**：口部有价值但碗(0.85)与盘(0.90)太近、颈仅两种器型有值、足各器型差距小(0.35~0.60)，单独权重过高反而引入噪声。

4. **数据验证支撑**：在18个测试案例中，肩+腹30%各的权重使所有案例正确匹配，没有产生混淆。

### 2.4 原始加权误差

```
RawShapeError = Σ( w_k × |P_k − T_k| )
              = 0.15×|P_m − T_m| + 0.15×|P_n − T_n| + 0.30×|P_s − T_s| + 0.30×|P_b − T_b| + 0.10×|P_f − T_f|

w = [Mouth=0.15, Neck=0.15, Shoulder=0.30, Belly=0.30, Foot=0.10]
```

**理论值域：** RawShapeError ∈ [0, 1.0]
- 0 = 完全匹配（各部位误差为0）
- 1.0 = 理论最大误差（每部位误差=1.0）

**实测分布（18个测试案例）：**

| 案例类别 | RawShapeError 范围 | 示例 |
|----------|-------------------|------|
| 完美匹配 | 0.0000 | TC-01~05 |
| 微偏 | 0.0160~0.0380 | TC-06~10, TC-18 |
| 边界混淆 | 0.0250~0.1305 | TC-11~15 |
| 极端输入 | 0.2800~0.3375 | TC-16~17 |

### 2.5 器型识别

```
BestMatch = argmin( RawShapeError(i) )  i ∈ {碗, 盘, 梅瓶, 玉壶春, 罐}

若出现平局（两个器型 RawShapeError 相等），取 Round-Robin 顺序中先匹配者。
```

> 当前 5 种器型的向量分布在各自区域足够离散，18个测试案例未出现平局。

## 3. ShapeScore 公式

### 3.1 正式公式

```
RawShapeError = Σ( w_k × |P_k − T_k| )   for k in 5 parts

ShapeScore = 100 × max( 0, 1 − RawShapeError / D_max )

其中 D_max = 0.35
```

### 3.2 参数解释

| 变量 | 含义 | 来源 |
|------|------|------|
| RawShapeError | 加权绝对误差和（0=完美，1=最差） | ShapeScore_验证系统.xlsx 算法 |
| D_max | 参考误差（分值为0的临界值） | 本规范 §3.3 |
| ShapeScore | 最终评分 0~100 | 本公式 |

### 3.3 D_max 的选择

D_max = 0.35，表示"加权绝对误差达到0.35时器型已完全不可辨识"。

| D_max 候选 | TC-14(边界极值0.1305) | TC-16(极端0.28) | 评价 |
|------------|----------------------|-----------------|------|
| 0.25 | 47.8 (D) | 0.0 (E) | 过严，边界案例惩罚过重 |
| 0.30 | 56.5 (C) | 6.7 (E) | 偏严 |
| **0.35** | **62.7 (C)** | **20.0 (E)** | **平衡** |
| 0.40 | 67.4 (C) | 30.0 (D) | 偏松，极端输入仍在D级 |

**选择 0.35 的理由：**

- 微偏案例(0.02~0.04)落A级(89~94)，与观众直觉一致
- 边界案例(0.06~0.13)落B~C级(62~83)，区分不同混淆程度
- 极端输入(0.28)落E级(20)，明确标识为"失败"
- 理论最大误差1.0的35%，对应"器型剖面严重变形"的视觉阈值

### 3.4 数学性质

| 性质 | 说明 |
|------|------|
| **单调性** | RawShapeError 递增 → ShapeScore 严格递减（误差越大分数越低） |
| **连续性** | 线性函数，无跳变点，微小的拉坯调整产生连续的分数变化 |
| **稳定性** | 对D_max不敏感（D_max±0.05范围内等级变化不超过1级） |
| **可扩展性** | 新增器型只需添加模板向量；V2.0可扩展为曲线评分 |
| **值域保证** | max(0, ...)确保无负数；D_max确保0~100全范围覆盖 |

### 3.5 单项部位得分

用于UI层向玩家展示各部位表现：

```
PartScore_k = 100 × ( 1 − |P_k − T_k| )
```

各部位独立评分0~100，供UI层雷达图/进度条使用。

## 4. 等级划分

统一采用 S/A/B/C/D/E 六级评级，**与 GlazeScore、FireScore 完全一致**。

| 分数区间 | 等级 | 说明 | RawShapeError 对应范围 |
|----------|------|------|----------------------|
| 95 ~ 100 | **S** | 完美 | 0.0000 ~ 0.0175 |
| 85 ~ 94 | **A** | 优等 | 0.0175 ~ 0.0525 |
| 70 ~ 84 | **B** | 良好 | 0.0525 ~ 0.1050 |
| 50 ~ 69 | **C** | 及格 | 0.1050 ~ 0.1750 |
| 30 ~ 49 | **D** | 偏差 | 0.1750 ~ 0.2450 |
| 0 ~ 29 | **E** | 失败 | 0.2450 ~ 0.3500+ |

> 禁止在评分系统内部使用中文品阶（贡品/珍品/上品/良品/次品/废品）。中文品阶仅作为 UI 展示层映射。

## 6. 优化扩展方向（不进入MVP）

### 6.1 曲线拟合修正项（V2.0）

当前 ShapeScore 仅基于 5 个控制点的比例值（点对点比较），属于**离散评分**。

V2.0 可扩展为**连续评分**：

```
AdvancedShapeScore = α × ProportionScore + (1−α) × ContourScore

ContourScore = 基于曲线相似度的评分（Hausdorff距离 / Frechet距离）
```

| 版本 | 评分基础 | 评估内容 |
|------|----------|----------|
| V1.0 (当前) | 5控制点比例 | 关键部位比例匹配 |
| V2.0 (未来) | + 轮廓曲线 | 曲线平滑度、过渡自然度 |

> V2.0 需要额外的曲线拟合数学库和性能分析，不进入MVP。

### 6.2 器型难度修正

利用 `Difficulty` 字段（1~5）：

**原则：Difficulty 不影响 ShapeScore 分值。**

| 用途 | 影响 |
|------|------|
| 订单收益 | 高难度器型 = 高金币/声望倍率 |
| 解锁条件 | Difficulty决定UnlockLevel |
| 经验值 | 高难度器型 = 更多经验 |
| ShapeScore | **不受影响** |

> 理由：ShapeScore是"器型做得好不好"的客观度量，不能被难度扭曲。一个歪的碗(S级简单器型)和一个歪的梅瓶(C级困难器型)，都应该得到与其拉坯精度对应的分数。

### 6.3 特殊器型权重建议（仅分析，不修改公式）

| 器型 | 特征 | 是否需要特殊权重 |
|------|------|-----------------|
| **碗** | 颈=肩=0，口大腹深 | **不需要**。特征简洁，现有权重完全覆盖 |
| **盘** | 与碗类似但腹浅(0.30) | **不需要**。腹权重30%已能区分碗/盘 |
| **梅瓶** | 口极小(0.15)，肩极大(0.85) | **不需要**。肩30%+口15%已充分捕捉 |
| **玉壶春** | 颈最长(0.55)，五项均非零 | **不需要**。唯一全维度器型，15%颈权重已足够 |
| **罐** | 腹最大(0.80)，口中等 | **不需要**。腹30%已充分捕捉 |

**结论**：当前权重 [0.15, 0.15, 0.30, 0.30, 0.10] 对5种器型均适用，18个测试案例全部正确识别。不需要特殊器型权重。

## 7. Unity 实现架构（非代码）

### 7.1 输入/输出

```
ShapeCalculator
├── 输入: PlayerShapeData
│   ├── float mouthRatio
│   ├── float neckRatio
│   ├── float shoulderRatio
│   ├── float bellyRatio
│   └── float footRatio
│
├── 输出: ShapeScoreResult
│   ├── float overallScore       (0~100)
│   ├── string grade             (S/A/B/C/D/E)
│   ├── string matchedShapeID    (SHAPE_001~005)
│   ├── float matchedShapeName
│   ├── float[] partScores[5]    (各部位0~100)
│   └── float rawError           (原始加权误差)
```

### 7.2 数据流

```
玩家拖拽5控制点
    │
    ▼
PlayerShapeData {mouth, neck, shoulder, belly, foot}
    │
    ▼
ShapeCalculator.Calculate(playerData)
    │
    ├── 读取 ShapeConfig 模板库 (5条)
    ├── 计算 5 个 RawShapeError(i)
    ├── 取 min → 确定最佳匹配器型
    ├── ShapeScore = 100 × max(0, 1 - RawShapeError / 0.35)
    ├── 计算 5 个 PartScore_k
    └── 确定 Grade (S/A/B/C/D/E)
    │
    ▼
ShapeScoreResult { overallScore, grade, matchedShape, partScores }
    │
    ├──→ UI层：雷达图展示5部位得分
    ├──→ Order系统：判断是否完成订单
    └──→ 经验系统：Difficulty加权经验值
```

### 7.3 依赖关系

```
ShapeCalculator
    ├── 依赖: ShapeSO (ScriptableObject, 模板数据)
    │         └── ShapeConfig (数据源)
    ├── 无依赖: GlazeScore
    ├── 无依赖: FireScore
    └── 被依赖: OrderValidator (判断订单是否完成)
```

## 8. 与其它评分系统的关系

### 8.1 三系统职责边界

| 系统 | 评估对象 | 输入 | 时机 | 问题 |
|------|----------|------|------|------|
| **ShapeScore** | 器型拉坯 | 5部位比例值 | 拉坯完成时 | "器型像不像？" |
| **GlazeScore** | 釉料配方 | Cu/Fe/Co 三元素 | 配方提交时 | "配方对不对？" |
| **FireScore** | 烧窑工艺 | 温度/时间/火焰 | 烧窑完成时 | "烧得好不好？" |

### 8.2 避免重复评分

| 边界 | 处理方式 |
|------|----------|
| ShapeScore ↔ GlazeScore | 完全独立，无重叠。器型与釉料是不同维度的评价 |
| ShapeScore ↔ FireScore | 完全独立。拉坯质量与烧窑质量是先后两个不同阶段 |
| GlazeScore ↔ FireScore | 独立但有因果链。GlazeScore决定"目标釉色是什么"，FireScore评估"是否成功烧出该釉色" |
| ShapeScore + 订单 | 订单验证：ShapeScore ≥ 阈值 + GlazeScore ≥ 阈值 + ShapeID匹配 + GlazeID匹配 → 订单完成 |

### 8.3 最终成品评价（未来 Phase）

```
FinalScore = (ShapeScore × GlazeScore × FireScore)^(1/3)
           = 三系统几何平均

或

FinalScore = w1×ShapeScore + w2×GlazeScore + w3×FireScore
```

> 最终成品评价公式在 ResultCalculator Phase 设计，本阶段仅预留接口。

**RawShapeError (B9):**

```excel
=D$5*ABS(B2-D2) + E$5*ABS(B3-E2) + F$5*ABS(B4-F2) + G$5*ABS(B5-G2) + D$4*ABS(B6-D3)
```

**ShapeScore (B10):**

```excel
=MAX(0, 100*(1-B9/B8))
```
