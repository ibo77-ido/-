# 遮挡层级执行方案

## 目标

建立女主与以下对象之间稳定的 2.5D 遮挡关系：

- 订单区
- 釉料区
- 原料区
- 仓库区
- 器型区
- 烧制区
- V_Willow_01
- V_Willow_02

目标效果：

- 女主站在遮挡物后方时，被建筑、柜台、树冠等前景元素挡住。
- 女主站在遮挡物前方时，显示在遮挡物前面。
- 地面、阴影、底座纹理不错误遮挡女主脚部。
- 女主移动时层级变化平滑，不出现整张区域图突然压住角色的问题。

## 当前已确认前提

当前场景为：

```text
Assets/Phase9/Scenes/SampleScene.unity
```

目标对象在场景中是独立 GameObject + SpriteRenderer，不是只合入一张静态大图。

当前结构：

```text
静态层/订单区
静态层/釉料区
静态层/原料区
静态层/仓库区
静态层/器型区
静态层/烧制区
静态层/tree/V_Willow_01
静态层/tree/V_Willow_02
女主/VisualRoot
```

当前主要问题：

```text
女主/VisualRoot sortingOrder = 900
V_Willow_01/V_Willow_02 sortingOrder 约为 805/803
六个区域 sortingOrder 约为 1-4
```

因此女主现在几乎永远压在目标场景物件上方，无法形成正确遮挡关系。

## 总体方案

分两阶段执行：

1. 先建立动态深度排序系统，让女主、柳树、区域对象能按 XZ 平面深度切换前后关系。
2. 再把需要遮挡的区域拆成 Base + Occluder，避免整张区域图错误遮挡女主。

不建议继续手动调整固定 sortingOrder。固定 order 只能在单个位置看起来正确，女主移动后会失效。

## 排序规则

项目当前为 XZ 平面行走，相机俯视。使用 `z` 作为视觉深度：

```text
z 越小，越靠屏幕前方，sortingOrder 越高
z 越大，越靠屏幕后方，sortingOrder 越低
```

推荐公式：

```csharp
sortingOrder = baseOrder + Mathf.RoundToInt(-sortPointZ * sortScale) + manualOffset;
```

推荐参数：

```text
baseOrder = 10000
sortScale = 1000
```

原因：

- 地图 z 范围如果约为 -3 到 +3，order 差值约为 6000。
- 这个范围足够区分女主、建筑、柳树前后关系。
- 比当前 1-4、803、900 的混合固定值更容易管理。

建议 order 区间：

```text
0-999        地面、水体、道路
1000-1999    区域 Base 完整底图
8000-16000   女主、柳树、Occluder 动态排序
20000+       特效或强制前景
Canvas UI    继续使用 Canvas sortingOrder
```

## 排序基准点

不要直接使用 Sprite 中心点作为排序点。不同对象需要不同的基准点。

女主：

```text
使用脚底点
通常可使用 女主 transform.position.z
实际 Renderer 为 女主/VisualRoot
```

柳树：

```text
使用树干落地点
第一版可使用 Sprite bounds.min.z 或手动 SortPoint
如树冠导致遮挡异常，再拆树冠遮挡层
```

区域 Occluder：

```text
使用建筑、柜台、墙体、台面前缘作为排序基准线
不要使用整张图片中心点
```

区域 Base：

```text
固定低 order
不参与动态排序
```

## 脚本计划

新增两个运行时脚本：

```text
Assets/Phase9/Scripts/DepthSorting/DepthSortable.cs
Assets/Phase9/Scripts/DepthSorting/DepthSortManager.cs
```

`DepthSortable` 职责：

- 标记一个对象参与动态排序。
- 指定实际要修改的 Renderer。
- 指定排序基准点。
- 提供手动微调 offset。

建议字段：

```csharp
[SerializeField] private Renderer[] renderers;
[SerializeField] private Transform sortPoint;
[SerializeField] private float manualZOffset;
[SerializeField] private int manualOrderOffset;
[SerializeField] private bool staticOrder;
```

`DepthSortManager` 职责：

- 收集场景中的 `DepthSortable`。
- 在 LateUpdate 中按排序公式刷新 order。
- MVP 阶段可每帧刷新全部对象。
- 后续可优化为只有女主排序点变化时刷新。

MVP 每帧刷新是可以接受的，因为参与对象数量很少。

## 第一阶段：最小验证

先不拆图，先接入动态排序，验证方向和公式。

接入对象：

```text
女主
V_Willow_01
V_Willow_02
订单区
仓库区
```

验证目标：

- 女主走到柳树后方时，柳树压住女主。
- 女主走到柳树前方时，女主压住柳树。
- 女主靠近订单区、仓库区时，前后关系方向正确。

注意：

第一阶段允许区域整图参与排序，只用于验证排序方向。它不是最终效果。

## 第二阶段：区域拆层

区域最终结构：

```text
订单区
  订单区_Base
  订单区_Occluder
```

`Base`：

```text
使用完整原图
固定 sortingOrder，例如 1000
永远在女主下面
不参与动态排序
```

`Occluder`：

```text
使用透明 PNG
只保留会挡住女主的前景部分
参与动态排序
使用前景遮挡物前缘作为排序基准点
```

## 拆图原则

不是把整张区域图切碎，而是额外制作一张透明遮挡图。

保留到 Occluder 的内容：

- 屋檐
- 柜台前沿
- 墙体前半部分
- 架子
- 屏风
- 大桌面
- 树冠
- 树干前景部分
- 大型器物前景边缘

从 Occluder 中擦透明的内容：

- 地面
- 路面
- 阴影
- 底座投影
- 不会挡住女主脚部的背景纹理
- 远景装饰

## 区域处理优先级

优先拆：

```text
1. 订单区
2. 仓库区
3. 器型区
4. 釉料区
```

视效果决定是否拆：

```text
5. 烧制区
6. 原料区
```

判断标准：

```text
有屋檐、柜台、墙体、台面等明确会挡住女主的结构，就拆 Occluder。
如果是开放场地、地面材料、低矮物件，可以不拆，只保留 Base 或整体动态排序。
```

柳树处理：

```text
第一版整体参与动态排序。
如果出现树冠大面积错误遮挡脚部，再拆成 BaseShadow + Occluder。
```

## 单个区域拆层流程

以订单区为样板：

1. 找到订单区当前 Sprite 对应 PNG。
2. 复制一份，命名：

```text
订单区_Occluder.png
```

3. 在图片编辑工具中只保留会挡住女主的前景部分，其余区域擦成透明。
4. 导入 Unity，并保持和原图一致的设置：

```text
Texture Type: Sprite
Sprite Mode: Single
Pixels Per Unit: 与原图一致
Pivot: 与原图一致
Filter Mode: 与原图一致
Compression: 与原图一致或 None
Alpha Is Transparency: true
```

5. 在场景中把原订单区调整为容器结构：

```text
订单区
  订单区_Base
  订单区_Occluder
```

6. `订单区_Base` 和 `订单区_Occluder` 的 localPosition、localRotation、localScale 保持一致。
7. `订单区_Base` 固定低 order。
8. `订单区_Occluder` 挂 `DepthSortable`。
9. 给 `订单区_Occluder` 设置 SortPoint，位置放在遮挡物前缘。

## 验收标准

每个对象完成后，至少测试三个位置：

```text
女主站在遮挡物后方：Occluder 应该挡住女主
女主站在遮挡物前方：女主应该挡住 Occluder
女主站在遮挡物侧边：不应出现整张大图突然盖住女主
```

重点检查：

- 女主脚部是否被地面贴图错误盖住。
- 女主头发、身体是否错误穿过屋檐、柜台、树冠。
- 柳树树冠是否在正确位置遮挡女主。
- 女主移动时 sortingOrder 是否出现明显跳变。
- 不同区域之间是否出现 order 抢层。

## 手动调整 SortPoint

区域遮挡不是只靠扣图决定的。女主在同一堵墙附近时，可能出现两种情况：

```text
女主在墙前：女主应该压住墙体 Occluder
女主在墙后：墙体 Occluder 应该压住女主
```

这个切换由 `SortPoint` 的 z 位置决定。当前可直接在 Unity 场景里调整：

```text
静态层/订单区/订单区_Occluder/SortPoint
```

选中 `SortPoint` 后，Scene 视图中会显示一条黄色/青色辅助线。它就是这个 Occluder 的排序判断线。

调整规则：

```text
如果女主站在墙前却仍被墙盖住：
把 SortPoint 往屏幕后方拖一点，也就是增大 z。

如果女主站在墙后却没有被墙盖住：
把 SortPoint 往屏幕前方拖一点，也就是减小 z。
```

只需要移动 `SortPoint`，不要直接改 `订单区_Occluder` 的 transform。`订单区_Occluder` 和 `订单区_Base` 必须继续保持同位置、同缩放。

判断顺序：

1. 先确认 Occluder 图只包含会挡人的墙、柱、屋檐、家具、台阶前沿。
2. 再移动 `SortPoint`，让女主在墙前/墙后时排序切换正确。
3. 如果只有局部墙面少扣或多扣，再回到 PNG 里改透明区域。

常见现象：

```text
整片地面盖住女主：
Occluder 扣图错了，地面不应该进入遮挡层。

墙前女主被墙盖住：
SortPoint 太靠屏幕前，需要往后调。

墙后女主压住墙：
SortPoint 太靠屏幕后，需要往前调。

只有某个墙角、桌脚、柱子错：
优先局部修改 Occluder PNG，不要为了一个角把整条 SortPoint 拖歪。
```

运行菜单：

```text
Phase9/Depth Sorting/Bind SampleScene Sortables
```

该菜单会补齐 `DepthSortable`、Base 固定层级、Occluder 动态层级和 `SortPoint` 辅助线。已有的 `SortPoint` 会被保留，不会被重新覆盖到自动位置。

## 构建器和重建风险

当前目标对象虽然是独立 GameObject，但项目存在 Phase9 构建和修复脚本。

风险：

```text
如果后续重新运行地图构建器或场景规范化工具，手动拆出的 Base/Occluder 结构可能被覆盖或丢失。
```

建议：

```text
先手动完成订单区样板。
样板通过后，把 DepthSortable 绑定、Base/Occluder 结构、sortingOrder 区间写入 Editor Repair 或 Builder 流程。
```

推荐后续落点：

```text
Assets/Phase9/Scripts/Editor/
```

可新增编辑器工具：

```text
Phase9DepthSortingBinder.cs
```

职责：

- 查找女主、柳树、六个区域。
- 自动补齐 `DepthSortable`。
- 自动设置 Base 固定 order。
- 自动设置 Occluder 动态排序参数。
- 防止重建场景后遗漏绑定。

## 推荐执行顺序

1. 新增 `DepthSortable` 和 `DepthSortManager`。
2. 接入女主、V_Willow_01、V_Willow_02、订单区、仓库区整图。
3. 测试动态排序方向。
4. 调整 `baseOrder`、`sortScale`、manualOffset。
5. 选订单区制作第一张 `订单区_Occluder.png`。
6. 将订单区改为 `Base + Occluder`。
7. 验收订单区遮挡效果。
8. 按优先级处理仓库区、器型区、釉料区。
9. 视效果决定是否处理烧制区、原料区。
10. 将绑定流程沉淀到 Editor 工具，避免后续重建丢失。

## 最终建议结构

```text
静态层
  地面/水体/道路
  区域底图层
    订单区_Base
    釉料区_Base
    原料区_Base
    仓库区_Base
    器型区_Base
    烧制区_Base
  动态遮挡层
    订单区_Occluder
    釉料区_Occluder
    原料区_Occluder
    仓库区_Occluder
    器型区_Occluder
    烧制区_Occluder
    V_Willow_01
    V_Willow_02

女主
  VisualRoot
```

实际落地时，不强制一次性创建所有 Occluder。优先保证排序系统正确，再逐个补遮挡层。
