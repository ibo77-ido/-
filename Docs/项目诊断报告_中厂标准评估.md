# 景德镇·窑火千年 — 项目全面诊断报告

> 评估视角：中型游戏厂商（中厂）开发标准  
> 评估对象：3 人团队 Unity 2.5D 制瓷题材游戏  
> 评估日期：2026-06-22  
> 评估范围：代码架构（159 个 .cs）、策划文档（200+ 份 .md）、美术资源（1071 个 .png / 839MB）

---

## 核心结论（先给答案）

**架构值得继续做，但必须"局部重构 + 全面规范"，不能原样继续，也不必推倒重做。**

理由：
1. **底层资产有价值**：Phase3 玩法层的数值模型（真实博物馆数据库 + 数学评分公式 + 8 阶段烧制工艺）达到准专业水准，这是项目最核心的竞争力，重做会丢失这些不可复现的知识沉淀。
2. **架构意图正确**：分层设计（世界层 / 桥接层 / 玩法层 / HUD 层）方向没错，Phase3 通过 `IGameplayProgressionAuthority` 接口解耦是教科书级做法。问题出在"上层没守住纪律"，而非"地基选错了"。
3. **真正需要重做的只有两处**：Phase9/Phase8 的双桥接必须二选一统一；Phase10 的反射穿透必须改成正式接口。其余都是"补规范 + 清理"的优化工作。
4. **美术管线必须从零建立**：目前几乎没有任何美术规范（无图集、无压缩标准、无命名约定），但这不等于美术资源要重做——资源本身可用，缺的是"管线"。

**一句话：保留地基和承重墙，拆除违建，补上缺失的管线。**

## 二、策划：不足与改进

### 现状评分：6.5 / 10

### 2.1 做得好的地方（要保留）

- **数值设计是全项目最专业的部分（9/10）**：器型/釉色数据库每条都标注了故宫博物院藏品编号和学术来源；评分系统有完整数学模型（ShapeScore / GlazeScore / FireScore 三段加权），含敏感性分析和风险表；烧制配置定义了 8 个工艺阶段 + 4 个火焰切换点 + 11 种缺陷，与真实景德镇柴窑工艺高度吻合。

### 3.3 给程序的建议

你的分层意图是对的，Phase3 的接口解耦证明你有架构意识。但**架构不是画在文档里的图，而是编译器能强制执行的约束**。当前分层只靠开发者自律维持，一旦换人、一旦赶进度，纪律就会崩塌（Phase11 就是例子）。**asmdef 程序集隔离是你最该补的一件事**——它让"违规引用"从"运行时崩溃"变成"编译报错"，是从根源解决问题的手段。

另外，God Object 和反射是"省事"的产物，但它们会让项目越往后越难改。**2774 行的 DialogueController 不是能力的证明，而是技术债的积累。**

## 五、架构决策：优化而非重做

### 5.1 为什么不推倒重做？

| 考量 | 结论 |
|------|------|
| 核心资产是否可复用？ | **是**。数值数据库（真实博物馆出处）、评分数学模型、8 阶段烧制工艺配置、6 个水墨 Shader——这些重做会丢失不可复现的知识沉淀 |
| 架构方向是否正确？ | **是**。分层设计方向正确，Phase3 接口解耦是教科书级做法。问题在执行纪律，不在设计意图 |
| 重做成本是否可接受？ | **否**。3 人团队重做等于回到零点，而当前已有可玩闭环（Phase3 玩法 + Phase6 世界 + Phase9 视觉） |
| 是否有不可修复的结构性缺陷？ | **没有**。所有问题（双桥接、HUD 侵入、反射、God Object）都是"可以拆解重构"的局部问题，不是"地基选错"的全局问题 |

### 5.2 为什么也不能原样继续？

当前状态如果原样继续，会面临：
- **改一处崩三处**：反射 + FindObjectOfType + 双桥接导致改动牵连面极广，任何重构都伴随运行时崩溃风险
- **包体失控**：839MB 且无合批，移动端根本无法发布
- **协作崩溃**：无 asmdef 隔离 + 命名混乱 + 文档碎片化，新成员上手成本极高
- **叙事断链**：StoryMain.md 丢失意味着剧情无法追溯，后续内容创作失去基准

### 5.3 推荐方案：局部重构 + 全面规范

分三个阶段，**保留地基和承重墙，拆除违建，补上管线**：

#### 阶段一：止血（2 周，P0）
1. 找回并版本化 StoryMain.md（策划）
2. 统一桥接层：Phase8/Phase9 二选一，删除冗余桥接（程序）
3. Phase11 HUD 改为事件/接口通信，删除硬引用（程序）
4. 创建 Sprite Atlas，按 4 类分包（美术）
5. 统一 PNG 导入设置（maxTextureSize 2048 + Crunch），包体降至 500MB 以内（美术）
6. 清理 Assets 内调试文件（Screenshots / Temp_ 场景）（工程）

#### 阶段二：加固（3 周，P1）
7. 为每个 Phase 创建 asmdef，编译期强制分层（程序）
8. 拆分 God Object（P10DialogueController / Phase9InteractionBridge / FiringPanelController）（程序）
9. Phase10 反射改为正式接口（程序）
10. Prefab 重构为 LogicRoot/ArtRoot 分离（美术/程序）
11. Shader 迁移到 HLSLPROGRAM（美术）
12. 补写统一 GDD + 经济循环设计（策划）
13. FindObjectOfType 改为注入（程序）

#### 阶段三：打磨（持续，P2-P3）
14. 叙事-玩法打通（策划/程序）
15. MP4 转 Sprite Animation（美术）

## 六、关于开发模式的坦诚观察

基于 Git 历史、决策记录和文档风格的分析，项目实际呈现的是**"1 名设计者 + AI Agent"的开发模式**，而非 PPT 所述的 3 人协作：

- AGENTS.md 整份文档是"AI Agent 执行规范"，run.md 是"AI 任务执行模板"
- DECISIONS.md 的决策都是"用户批准"——单一决策者
- Git 提交风格高度一致——单一提交者
- 文档膨胀（Phase10 的 56 份 .md）和反射伪解耦，是 AI 生成代码的典型特征

**这不是问题本身，但需要正视**：
- 如果确实逐步引入更多人类成员，当前的 asmdef 缺失和命名混乱会成为协作障碍
- 如果继续"1 人 + AI"模式，AGENTS 工作流体系需精简（后期已开始退化，证明不可持续）
- AI 擅长生成"看起来正确"的代码，但会埋下反射、God Object、伪解耦等隐患——这正是当前程序问题的根源
